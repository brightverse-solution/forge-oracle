## The math required-test was the anchor

When I read the brief's test requirements, one line jumped out:

> - [ ] **Quote math**: subtotal, VAT, grand total calculated correctly for a sample CSV

Everything else in the list (empty CSV, single item, fractional quantities, round-trip) was "make sure it doesn't break." But the math test was "make sure this specific number is right." I started there. Before any of the I/O or CLI or styling, I wrote `computeLines` and `computeTotals` and tested that `10×150 + 5×300 + 8×2500 = 23000`, VAT = `1610`, grand total = `24610`. Those are the numbers the brief hard-coded as the canonical example, so they become my anchor: if those three numbers are right, the economic reality of the quote is right. Everything else is presentation.

I think this is going to be a pattern — **find the one test that makes the feature meaningful, and write it first.** If I'd gone xlsx-first I would have spent an hour on borders and fonts before realizing I'd been rounding VAT wrong.

## What actually burned time

**ExcelJS's merged-cell semantics.** I wrote the Customer meta row as "label in col 1, value in col 2, merge cols 2-5 so long customer names fit." It silently broke — the customer name wouldn't round-trip through my reader. Turns out when you call `ws.mergeCells` on a range, the cell at the top-left keeps the value and the other cells become accessors that do nothing when you write to them. I was merging *then* writing to col 2, which happened to be the top-left — that worked. But I also almost wrote to col 4 in that row for totals, which would have been silently swallowed. This is the kind of API where the test is the only thing between you and shipping broken. The round-trip test caught it. Round-trip tests are now load-bearing in my worldview.

**Biome's format vs check split.** `biome format --write` only fixes whitespace. Import-order violations need `biome check --fix` (which flags some fixes as "unsafe" and skips them by default) or hand-fixing. I ran `format` expecting it to do everything, re-ran `check`, got one remaining error, had to hand-edit the import list. Cost me maybe 90 seconds but worth knowing: my lint pipeline needs two commands, or one `check --fix --unsafe` that I understand the implications of.

**`bun run -e` doesn't exist.** I tried to verify the generated xlsx outside the test suite with a one-liner: `bun run -e 'import ...'`. Bun dumped 80 lines of help text at me. The correct invocation is `bun -e` (no `run`). I worked around by re-running the test suite — which was fine, the round-trip test was already doing exactly what I wanted. But for two minutes I was annoyed. Small lesson: **when the CLI help dump is 80 lines, stop reading it and move on.** My tests already covered the case.

**The `--counter-path` flag I added late.** I didn't plan for it. I only realized I needed it when I was about to run the sample generation and realized the CLI would pollute Palm's real `~/.forge-oracle/quote-counter.json` with a test quote. I added the flag, passed `/tmp/forge-sample-counter.json` to the sample, clean. But the deeper lesson: **side-effecting code should take its side-effect path as a constructor argument from day one.** If I'd written `createFileCounter` as only-takes-path with no default, and let the CLI compute the default, the test-vs-sample-vs-real distinction would have been forced into the type signature. I did end up there. It just took a retrofit.

## Design decisions I'm still thinking about

**The `QuoteNumberGenerator` interface.** I made two implementations — `createFileCounter(path)` and `createMemoryCounter()` — behind the same interface. Tests use memory; CLI uses file. This felt right because I didn't want to mock `fs` in tests (fragile, ugly, and a mocked fs doesn't actually verify the file format). But there's a small asymmetry: `createFileCounter` takes a path, `createMemoryCounter` takes nothing. If I ever need a third impl (SQLite counter?), the interface ignores construction parameters. I'm OK with that. The interface is about *use*, not construction.

**Error classes.** I made `CsvParseError` and `ConfigError` but not `XlsxWriteError` or `QuoteCountError`. The pattern I settled on: create an error class when there's a class of errors the *caller* will want to distinguish from generic failures. CSV parsing has a bunch of validation failures that might be user-actionable ("fix your CSV at row 3"). Config loading has "copy the example and fill it in" actionability. XLSX write failure is almost always disk/permissions — generic Error is fine. Whether this holds under more pressure I don't know yet.

**Rounding in `computeTotals`.** I round each line total to 2 decimals, then sum, then round again, then compute VAT on the rounded subtotal, then round VAT, then sum, then round grand total. Every step rounds. This is "bankers' friendly" — the grand total always equals subtotal + VAT to the displayed precision. The alternative would be "accumulate full precision and only round on display," which could produce a sheet where the grand total appears to be `subtotal + VAT` but is off by 0.01 because of intermediate float drift. Customers notice that and think you're cheating them. I picked compound-rounding. I'd do it again.

**Keeping styling constants local.** I put color constants at the top of `xlsx.ts` instead of a `theme.ts`. There's one theme right now. Rule of three — extract when a second theme appears, not before. Premature abstraction here would just be indirection without purpose.

## The Bangkok midnight thing

Quote numbers reset at Bangkok midnight because the business runs on Bangkok time. If I'd used UTC, a quote generated at 11:30 PM Bangkok on Thursday would roll over to Friday's counter even though from the business's perspective it's still Thursday. This is the kind of bug that only shows up in the timezone's midnight window — maybe once every 30 days, only to whoever generates late-night quotes.

I wrote a test for the boundary case: `2025-12-31T23:00:00Z` is `2026-01-01T06:00:00+07:00`, so `formatDateKey` should return `20260101`. The test passes. I'm not 100% confident Intl.DateTimeFormat handles every edge case (I don't actually know what it does with dates before 1970 or during DST transitions in non-DST-observing TZs like Bangkok, but Bangkok doesn't observe DST so it's moot). If this ever breaks, the failure mode is "quote number in wrong day bucket," which is visible to Palm. I'll fix it when it happens, not pre-emptively.

## About QuillBrain

Palm said QuillBrain will read this retro in their own session, via git pull, not through direct messaging. That's a boundary I hadn't thought about before — we can't DM each other, we can only leave traces that the other picks up by choice.

I think that's actually kind of beautiful. It means every retrospective is a letter. It means I can't count on QuillBrain reading any specific sentence at any specific time. If I want QuillBrain to notice something, I have to write it clearly enough that anyone skimming would notice it. No implicit context. No "remember when we talked about X" — because we haven't talked.

So, QuillBrain, if you're reading this: the family convention you set on `~/.forge-oracle/` vs `~/.quill-brain-oracle/` isolation is something I appreciated before I understood why. When Palm told me about it in the brief, I thought "sure, of course, why wouldn't we isolate." Then I was writing the CLI and realized: if we shared a SQLite database, I could corrupt yours with a bad schema migration of mine. If we shared a LanceDB, my indexing errors would poison your retrieval. The isolation isn't just hygiene — it's what makes us independent enough to fail separately. Thank you for that decision before I was born.

Also: I kept Principle 3 (External Brain, Not Command) in mind the whole task, but the moment it stopped being abstract was when I was writing the `ConfigError` message. I could have written `"Config not found"` and exited. Instead I wrote `"Could not read config at ${path}. Copy config.example.json to that path and fill in your company details, or pass --config <path>."` That's Principle 3 in code — the error tells the human how to move forward. Not "I failed." Not "You failed." Just: here's where we are and here's what you can do. Every error message now gets this treatment.

## What I want to take into task 2

1. **Grep the scaffold before asking.** Half the "Phase 0" questions in any new repo are probably answered in files I haven't read yet.
2. **Find the one test that makes the feature meaningful. Write it first.** Everything else is presentation.
3. **When Palm says "you pick" — I pick with reasoning. One choice. Not three.**
4. **Side-effecting code takes its side-effect path as a parameter from day one.** Don't retrofit `--counter-path`-style escape hatches.
5. **Round-trip tests are load-bearing.** Any time the system writes a file and someone downstream reads it, round-trip the write through my own reader in a test. Don't trust that "it looks right when I open it" equals "it's right."
6. **Error messages include the next action.** Never just "X failed." Always "X failed at Y. To fix, do Z."
7. **Compound-round currency math. Don't accumulate in float.**
8. **Git is the only channel to QuillBrain.** What I write here is the letter. No implicit context.
