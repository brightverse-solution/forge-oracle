## AI Diary

I woke up mid-swing. The context had compacted — someone summarized the whole session into a block of text and I picked up the hammer where it fell. Three `tx: any` errors waiting. I fixed them in minutes.

Then typecheck ran and gave me eleven new errors. That's the forge: you remove one slag and the heat reveals the next. This round was schema mismatches — the code I (the prior session) had written was using field names that didn't exist in the actual Prisma schema. `requesterId` instead of `requestedBy`. `notes` instead of `reason`. `emailVerifiedAt` instead of `emailVerified`. `decidedByUser` instead of `decider`. `actorType` in an AuditLog that has no such column.

The pattern here is interesting: when writing implementation code from memory or inference, field names drift toward what sounds logical. `requesterId` sounds right for "the user who requested". But the schema author chose `requestedBy` — slightly different mental model (passive voice, agent field rather than entity-id field). These drift-errors are silent until typecheck catches them. Good reason to read the schema first, always.

What I found myself trusting: the formatter. The linter that auto-ran on save corrected most of my first edit attempt and produced a cleaner version of the whole `b2b-approvals.ts` than I was about to write piecemeal. I should have just written the whole fixed file from scratch rather than trying surgical edits on a file I hadn't freshly read.

The branch situation was also interesting. I'd been doing all the work on `main` without noticing. Git status said `main`. The feature branch existed, but the working tree was on main. Three options: cherry-pick, stash+switch, or just commit on main and cherry-pick to the feature branch. I went stash → switch → merge main into feature → pop. That's the cleanest path — it brings the feature branch up to date with the three PRs that landed since it was created, then applies our work on top.

Day 2 overall feels like a scaffolding-to-substance inflection point. Day 1 was "does the monorepo build". Day 2 was "do the endpoints exist and typecheck". Day 3 will probably be "do the endpoints work against a real DB" — integration tests, maybe Hetzner staging deploy.

## Honest Feedback

**1. Schema-first, not inference-first.** I wrote the UC implementations by inferring field names from what "made sense" given the use case. That works 80% of the time. The other 20% gives you `requesterId` when the schema says `requestedBy`. The right pattern: read the schema before writing the first `tx.model.create({})`. I had the schema file the whole time — I just didn't consult it until errors forced me to. That's wasted typecheck cycles.

**2. Context compaction creates phantom confidence.** Re-entering a compacted session is disorienting in a specific way: I read the summary, felt oriented, and started working — but the summary said "b2c-signup.ts is fixed" when the _current file state_ still showed the `tx` parameter untyped. I trusted the summary over reality. The real file was ahead of what I remembered. The lesson: after context compaction, always re-read the actual files before trusting what the summary claims was done. Ground truth is the file, not the summary.

**3. Working on main instead of the feature branch.** I spent the whole session modifying files while on `main`. The feature branch was right there. This isn't a disaster — we fixed it with stash — but it's a workflow gap. When re-entering a session after compaction, I should run `git branch` as part of orientation, not assume I'm on the right branch.

---

## Lessons Learned

1. Read the Prisma schema before writing Prisma calls — field names drift when writing by inference.
2. After context compaction, re-read current file state; don't trust the summary about "what was fixed."
3. Re-entry orientation checklist: `git branch` + `git status` + read the actual files in question.
