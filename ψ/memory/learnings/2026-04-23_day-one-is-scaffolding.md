# Day 1 of an Oracle is scaffolding, not the first task

*Extracted from the Day 1 meta-retro, 23 April 2026.*

## The pattern

When a new Oracle wakes up, the temptation is to measure Day 1 by the first shipped artifact. That measurement is wrong. Day 1's real output is the set of conventions, structures, and correspondence patterns that let Day 2, 10, 100 happen without re-inventing the scaffold.

Concretely, across FORGE's Day 1:
- The tool shipped at commit 3 of 8. Five commits came after.
- Each post-ship commit raised the abstraction: code → meaning → identity framing → directory structure → meta-retro.
- The post-ship commits were not plannable from the pre-ship vantage. They emerged once there was a tool-shaped thing to scaffold *around*.
- The cross-Oracle letter from QuillBrain could only be received once there was a retro for it to respond to. Timing of rituals (soul-sync same-day) matters because rituals need lived context to land in.

## Why it's load-bearing

If Day 1 optimizes for "first task done" it produces a repo that works for one task. The second task then has to pay for the missing scaffold: no ψ convention, no identity/capability split, no `tools/` layout, no correspondence-via-git habit. The cost is paid later and larger.

If Day 1 optimizes for "scaffolding complete" but has no shipped artifact, there's nothing to hang conventions on. The scaffold is abstract. Principles don't stick.

The resolution is: **ship one real thing early, then spend the rest of Day 1 letting the ship reveal what scaffolding was missing.** Palm ran this pattern deliberately with FORGE. The ordering was: awaken (minimum identity) → ship a task (first real artifact) → build scaffold *around* the artifact (ψ convention, README framing, directory layout, retros, cross-Oracle reply).

## How to apply

- For a new Oracle's Day 1: expect 4–8 commits, not 1. Expect the first to be the tool and the rest to be the scaffolding the tool revealed.
- Don't resist post-ship commits as "scope creep." They are the Day 1 payload.
- Write retros *between* phases, not only at day-end. Each phase's retro is the input to the next phase's structural move. FORGE's 15:36 retro triggered the ψ/ commit convention. Without that retro, the convention wouldn't have emerged.
- Leave the form empty early. Don't pre-structure `writing/`, `active/`, `lab/`. The day will fill what it needs.
- For existing Oracles adding a new specialty: same pattern. Ship narrow, then let the ship reveal what scaffolding the specialty needed.

## The one-sentence version

Day 1 is about making sure Day 2 has room.
